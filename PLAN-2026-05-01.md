# Shiro — Strategic Plan (Remote + Coding Orchestration)
**Date:** 2026-05-01
**Inspirations:** Anthropic Paperclip, Claude Code (CLI), generic ChatOps bots
**Vision:** Shiro = personal coding co-pilot you can talk to from anywhere (Telegram / iPhone) that orchestrates your local dev environment (VS Code + Claude Code CLI) on your behalf.
**Note:** Supplement to existing `PLAN.md`. This doc focuses on remote access + coding-orchestration roadmap requested 2026-05-01.

## PART A — Remote Access from Mobile

### Goal
Send a message from phone → Shiro on Mac receives it → routes it to the right agent → replies on the same channel. Bidirectional.

### Current state (verified 2026-05-01)
- `TelegramRelay.swift` is **one-way + callbacks only**:
  - `notify()` and `sendApprovalCard()` push out
  - `handleUpdate()` only handles approval-button taps, not free-form text
- No HTTP server, no PWA, no iMessage hook.
- The infrastructure for remote chat doesn't exist yet — only remote *approval* does.

### Transport options — ranked

| # | Option                       | Effort | UX on iPhone     | Verdict |
|---|------------------------------|--------|------------------|---------|
| 1 | **Telegram bidirectional**   | S      | Native Telegram  | ✅ Ship first — relay already wired |
| 2 | **iOS Shortcuts → HTTP**     | M      | Home screen tap  | ✅ Ship second — adds Siri voice |
| 3 | **Tailscale tunnel**         | S      | (infra)          | ✅ Required for #2 to work outside LAN |
| 4 | **PWA companion**            | L      | Browser-based    | ⚠️ Defer — duplicate of Telegram UX |
| 5 | **iMessage / Messages.app**  | M      | Native iMessage  | ⚠️ Brittle (AppleScript breaks on macOS upgrades) |
| 6 | **Matrix bot (E2E)**         | L      | Element app      | ⚠️ Defer — Telegram Secret Chats cover most paranoia |
| 7 | **Signal / WhatsApp**        | XL     | Native           | ❌ No proper bot API for Signal; WhatsApp Cloud API is enterprise-priced |
| 8 | **Discord bot**              | M      | Native Discord   | ❌ Privacy: your prompts on Discord servers |

### Recommended stack
**Telegram (full-duplex chat) + iOS Shortcuts (voice + widgets) + Tailscale (network).**

- **Telegram** — best in-class bot API, free, encrypted in transit, push notifications, image support
- **iOS Shortcuts** — "Hey Siri, ask Shiro to refactor auth" → reply spoken back
- **Tailscale** — zero-config WireGuard mesh; phone reaches Mac without port-forwarding or ngrok tokens

### Architecture — `RemoteInbox` abstraction

Transport-agnostic envelope so all remote channels feed one queue:

```
                 ┌─────────────────┐
                 │  RemoteMessage  │  { sender, channel, text, attachments, msgId }
                 └─────────────────┘
                          ▲
       ┌──────────────────┼──────────────────┐
       │                  │                  │
┌──────────────┐  ┌──────────────┐  ┌────────────────┐
│ Telegram     │  │ HTTP Server  │  │ (future) PWA   │
│ Relay        │  │ (Shortcuts)  │  │ WebSocket      │
└──────────────┘  └──────────────┘  └────────────────┘
       │                  │                  │
       └────────┬─────────┴──────────┬───────┘
                ▼                    ▼
         ┌──────────────────────────────┐
         │  RemoteInbox (MainActor)     │
         │  - dedupe by msgId           │
         │  - route to BridgeRouter     │
         │  - track conversation state  │
         └──────────────────────────────┘
                       │
                       ▼
              ┌─────────────────┐
              │  BridgeRouter   │  (existing — picks ACP / ClaudeCode / etc.)
              └─────────────────┘
```

### New / changed files
- **NEW** `Sources/Bridge/RemoteInbox.swift` — actor; conversation state per remote channel
- **NEW** `Sources/Bridge/HTTPRemoteServer.swift` — embedded `Network.framework` HTTP listener on `127.0.0.1:7421` (Tailscale-routable)
- **EDIT** `Sources/Bridge/TelegramRelay.swift` — handle non-callback `message.text` updates → forward to RemoteInbox

### Auth model
- **Telegram** — validate `chat_id` matches keychain `telegram_chat_id` (already done)
- **HTTP** — bearer token in keychain (`shiro_remote_token`); rotates from Settings tab
- **Tailscale** — relies on tailnet ACL; only your devices reach the Mac
- **Never expose to public internet.** Bind only to `127.0.0.1` + Tailscale IP.

### Wake / reachability
- ❌ Cold-sleep wake — hardware-dependent, fragile. Document, don't ship.
- ✅ Screen-locked Mac — works fine; Tailscale + Shiro process stay alive
- ✅ Caffeinate during remote work — `caffeinate -d -t 600` so screen doesn't sleep mid-task
- 🔄 WoL fallback — document setup but don't bake in (router-dependent)

### 3-phase rollout

**Phase 1 — Telegram chat (1 day)**
- Extend `TelegramRelay.handleUpdate` to handle `message.text`
- Pipe message text into `BridgeRouter.run(prompt:)`
- Stream agent reply back via `notify()`
- Add `/start`, `/stop`, `/model`, `/cancel` Telegram commands

**Phase 2 — HTTP + iOS Shortcuts (2 days)**
- Build `HTTPRemoteServer` with `POST /v1/prompt` + `GET /v1/status`
- Generate iOS Shortcut: "Ask Shiro" → text input → POST → speak reply
- Bind to localhost + Tailscale IP only
- Setup doc in `docs/REMOTE.md` (Tailscale install, Shortcut import URL)

**Phase 3 — Conversation continuity (1 day)**
- `remote_sessions` table → resume context across messages
- Per-conversation token budget (auto-summarize after N turns)
- Status indicator in main window: 🟢 Phone (last msg 2m ago)

## PART B — Coding Orchestration ("send task → opens VS Code + Claude Code")

### User story (your words)
> "Once I give it a task, it gives a prompt, but it should understand to open Claude. It should open VS Code and then open Claude — run Claude Code in CLI — and then I should be able to use it from there."

### Terminology clarification
There are two "Claude"s in your sentence and they aren't the same thing:
- **Claude desktop app** (claude.ai for Mac): GUI, no Swift-callable API. Can launch but can't pipe prompts in.
- **Claude Code CLI** (`claude` command): full programmatic control, headless mode (`-p`), streaming JSONL transcripts, hooks. **This is the right target.**

### Actual flow we'll build
1. You send a task from phone or floating bar
2. Shiro picks a project (asks if ambiguous)
3. Shiro creates an isolated git worktree for the task
4. Shiro opens VS Code on that worktree
5. Shiro opens an iTerm tab inside the worktree, running `claude "<refined-prompt>"`
6. Claude Code starts working
7. Either continue in iTerm on Mac, OR Shiro streams Claude Code's output back to Telegram so you watch from phone
8. When done, Shiro reports completion + diff summary + next steps

### What's possible (✅), hard (⚠️), not possible (❌)

| Capability | Status | Mechanism |
|---|---|---|
| Open VS Code at a path | ✅ | `code <path>` (already in `$PATH`) |
| Open VS Code on specific files | ✅ | `code -g <file>:<line>` |
| Open iTerm with command | ✅ | AppleScript `tell app "iTerm"` |
| Open Terminal.app with command | ✅ | AppleScript |
| Run `claude` headless, capture output | ✅ | `Process` + `claude -p "..." --output-format stream-json` |
| Stream Claude Code transcript live | ✅ | Tail `~/.claude/projects/<encoded>/<sid>.jsonl` |
| Drive Claude desktop app | ❌ | No API; UI scripting fragile + breaks per release |
| Drive Claude VS Code extension panel | ⚠️ | VS Code Commands URI works but limited surface |
| Inject prompts into running interactive `claude` session | ⚠️ | tmux send-keys works but breaks on multi-line + ANSI |
| Resume Claude Code session from phone | ✅ | Use `claude -c` (continue last session) |
| Wake Mac from cold sleep | ❌ | Hardware-dependent; document WoL but don't ship |

### Architecture — `CodingOrchestrator`

New module, sits next to `HooksEngine`:

```
Sources/Agent/CodingOrchestrator.swift
  ├── func plan(task: String) -> CodingPlan          // LLM call: structures task
  ├── func resolveProject(hint: String) -> Project   // fuzzy-match against workspaces
  ├── func createWorktree(_ project, branch) -> URL  // git worktree add
  ├── func openIDE(at: URL, files: [String])         // launches VS Code
  ├── func launchClaudeCode(at: URL, prompt, mode)   // headless or interactive
  └── func streamTranscript(sessionId) -> AsyncStream // tails JSONL
```

A `CodingPlan` is structured:

```swift
struct CodingPlan: Codable {
    let projectPath: URL
    let branchName: String          // auto: "shiro/<short-task-slug>"
    let prompt: String              // refined for Claude Code
    let mode: Mode                  // .interactive | .headless | .background
    let openInIDE: Bool
    let allowedTools: [String]?     // pass to claude --allowedTools
    let estimatedDurationMins: Int?
}
```

### New skill — `code_with_claude`

Registered in `SkillsRegistry`. Invoked by:
- Slash command: `/code <task>` (in floating bar / main window)
- Telegram: free text containing "code:" prefix or implicit detection
- iOS Shortcut: dedicated "Code with Shiro" shortcut

Skill logic:
1. If no project context, ask which project (Workspaces registry; LLM fuzzy match)
2. Run `plan(task:)` → produces a `CodingPlan`
3. Show plan back to user (Telegram preview / floating bar approval card)
4. On approve:
   - `git worktree add` (logged via `logError` audit if it fails)
   - Launch VS Code (if `openInIDE`)
   - Launch Claude Code in chosen mode
5. Stream progress:
   - **Headless** → tee Claude's stdout to chat
   - **Interactive** → confirm "started in iTerm tab — switch to Mac to continue"
   - **Background** → `--no-tty`, post final summary when done

### iTerm AppleScript template (proven pattern)

```applescript
tell application "iTerm"
    activate
    set newWindow to (create window with default profile)
    tell current session of newWindow
        write text "cd " & quoted form of "/path/to/worktree"
        write text "claude " & quoted form of prompt
    end tell
end tell
```

Wrap in Swift `Process` with `osascript -e ...`.

### 3-phase rollout

**Phase 1 — Manual orchestration (2 days)**
- `CodingOrchestrator` module + `code_with_claude` skill
- Slash command `/code` works from floating bar
- Opens VS Code + iTerm + claude interactive mode
- No streaming yet, no Telegram integration

**Phase 2 — Headless + Telegram (2 days)**
- Add `--mode headless` path: spawn claude with `-p`
- Tail JSONL transcript → format into chat-friendly chunks
- `/code` works from Telegram and streams back
- Add `/cancel` to kill running orchestration

**Phase 3 — Smart task understanding (3 days)**
- Plan-preview in approval card before launching
- Project auto-detection from natural language ("the shiro repo")
- Worktree cleanup `/code-cleanup` to prune merged branches
- Cost & time estimate before starting

### Soon — next quarter

15. **Per-project workspace presets** — `.shiro/workspace.toml` with model + allowed tools + memory scope
16. **Routine dry-run mode** — preview what a routine would do without executing

### Aspirational — bigger swings

17. **Multi-agent orchestration** — Shiro spawns N parallel Claude Code agents on different worktrees, monitors progress, merges PRs (Paperclip-style)
18. **Local LLM fallback** — route low-risk tasks to LM Studio (already configured per CLAUDE.md) → cost = $0
19. **Memory graph visualization** — `KnowledgeGraph` already exists; add interactive view
20. **Screen recording → routine** — capture 30s screen recording → vision model → derive a routine
21. **Cross-machine sync** — `tool_policies`, conversations, routines via iCloud Drive / CloudKit
22. **Phone-to-Mac handoff** — start task on phone via Telegram, finish on Mac (resume same Claude Code session)

### Won't do — and why

| Idea | Why not |
|---|---|
| Drive Claude desktop GUI via UI scripting | Brittle, breaks every Claude release. CLI is the right surface. |
| Wake Mac from full sleep over the internet | Hardware-dependent, requires WoL on router. Document, don't ship. |
| Self-host the LLM inside Shiro process | Bloats binary; we have LM Studio. Shell out. |
| Train custom embeddings model | Use OpenAI/Voyage — orders of magnitude cheaper than training |
| Build a full mobile app (Swift iOS) | Telegram + Shortcuts cover 90% with 5% of the effort |
| Discord / Slack bots | Privacy: prompts on third-party servers |
| Replicate Claude Code's whole feature set inside Shiro | Don't compete with Claude Code — orchestrate it |
| Multi-user / multi-tenant Shiro | Single-user-by-design; complexity tax not worth it |
